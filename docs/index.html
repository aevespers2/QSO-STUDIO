Product/UX charter candidate

Inspect complex QSO evidence without inheriting hidden authority.

QSO-STUDIO is a proposed human-review surface for genomes, state, messages, provenance, freeze points, experiments, release evidence, and payment records. Its current repository surface is documentation—not a verified application.

Current boundary: read-only review and proposal export. No runtime execution, unrestricted repository writes, credentials, autonomous approval, or production payment control.

Purpose

A review instrument, not a control plane.

Understand

Bring versioned object, genome, evidence, message, freeze, provenance, and release records into one coherent review context.

Compare

Inspect differences, validation outcomes, uncertainty, failures, and source hashes without rewriting the underlying evidence.

Propose

Prepare bounded review artifacts for an external human-approval workflow. A proposal never becomes execution merely because it was created in Studio.

Read-only evidence review

Validate first. Render safely. Export deliberately.

Select bundle→Verify version and hashes→Normalize and redact→Inspect evidence→Annotate→Export review artifact→External approval

Fail closed

Unsupported, stale, malformed, oversized, or hash-mismatched evidence remains rejected or visibly unresolved. Studio does not invent a compatible interpretation.

Preserve provenance

Every displayed value remains linked to its source bundle, repository, immutable commit, contract version, and transformation history.

Treat content as data

Imported text, markup, links, diagrams, and metadata are untrusted. They cannot execute scripts, commands, dynamic imports, or repository instructions.

Keep approval external

Annotations and proposals are distinct from source evidence and require a separately authorized workflow before any operational action.

Proposed information architecture

Each view answers a review question and carries no hidden operational power.

Object overview

Identity, genome reference, declared capabilities, state summary, and evidence freshness.

Evidence explorer

Sources, hashes, outcomes, uncertainty, contradictions, transformations, and attribution.

Message timeline

Bounded communications, ordering, causal links, limits, and protocol errors.

Freeze inspector

Snapshot hashes, stop reasons, runtime limits, rollback references, and integrity status.

Release view

Task-chain state, acceptance gates, artifacts, checksums, provenance, blockers, and approvals.

Payment evidence

Intent, authorization, allocation, receipt, dispute, environment, and reconciliation records—read-only and never a settlement control.

Authority boundaries

Display, proposal, approval, execution, writeback, and settlement remain separate.

Future integrations must be independently versioned, authenticated, least-privilege, auditable, and explicitly approved. No such integration is part of the current documentation candidate.

Accessibility and privacy

Evidence is only useful when people can perceive, navigate, and question it.

Multiple representations

Every visualization requires a structured text or table alternative. Status never depends on color alone.

Complete keyboard path

Navigation, inspection, filters, errors, annotations, and export must be operable without a pointer.

Privacy minimization

Personal, financial, and operationally sensitive metadata is minimized, classified, redacted, and excluded from public artifacts by default.

Recoverable errors

Validation failures identify the affected record, preserve original evidence, and offer a safe return to the review workflow.
